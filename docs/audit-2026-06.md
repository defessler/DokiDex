# DokiDex Codebase Audit Report
**Date:** 2026-06-23
**Scope:** First-party code across control/ (C# WPF + ASP.NET), serving/ (PowerShell + llama-swap), evals/, and setup.ps1

---

## Executive Summary

DokiDex is a well-intentioned single-developer project with genuine engineering discipline: pure/testable service functions extracted throughout, consistent graceful-degradation contracts, a real xUnit test suite with 563+ Fact methods, argv-array-based process launching (no shell-string injection anywhere), and a working eval gate. The data layer is sound.

Two structural debts dominate the risk profile and will compound sharply as features are added: `StudioHost.cs` is a 1,296-line file with a single 1,170-line `MapApi` method that owns routing, protocol logic, business rules, and infrastructure concerns simultaneously; and `index.html` is a 2,741-line monolithic SPA with all 12 views, all global state, and all polling in a single flat script tag. Everything else is secondary to these two. Security posture is generally good — the DNS-rebinding defense is solid — but a CSRF gap on the no-Origin path and one unvalidated LLM-generated file path in the chat tool loop are the two issues requiring immediate attention. The model serving layer has a concrete C#/PowerShell drift that silently breaks three gen kinds from the panel.

**Top 5 risks:**

1. CSRF via no-Origin HTML form POST reaches every state-changing API (delete gallery, queue generation, flip GPU mode)
2. `edit_image` tool accepts LLM-generated source paths that bypass `GalleryService.Resolve`, allowing arbitrary file reads
3. `GenRequest` missing `-Audio`/`-Engine` fields — `InfiniteTalk`, `LatentSync`, and `Speak` silently broken from the panel
4. `CUDA_VISIBLE_DEVICES=-1` set in parent process by `start-embed.ps1`, leaking into all subsequent doki.ps1 children
5. Zero SHA-256 verification on auto-downloaded model weights (embed model, SAM checkpoint, LLM candidates)

## Findings by Severity

### P1 — Fix Now

---

**[P1-1] CSRF middleware accepts no-Origin requests on all mutating verbs**
`control/Web/LocalSecurityMiddleware.cs:29` | Effort: S

The condition `!string.IsNullOrEmpty(origin)` short-circuits for HTML forms and `fetch(mode:'no-cors')`, both of which omit the Origin header. A malicious webpage can POST to `127.0.0.1:5111` to delete gallery items, queue generations, flip GPU mode, or write memory notes. DNS-rebinding is correctly blocked; this is a pure cross-site form-post vector.

**Fix:** At line 31, require Origin to be present *and* in AllowedHosts on every non-safe verb. Simplest implementation: deny when `string.IsNullOrEmpty(origin)` on POST/PUT/DELETE/PATCH, or add a `X-Requested-With: dokidex` custom header that the SPA always sends and the middleware requires. The inline SPA's own `fetch()` always provides Origin so this will not break legitimate traffic.

---

**[P1-2] `edit_image` tool stores LLM-generated source path in PendingGen without scope validation**
`control/Web/ChatTools.cs:487` | Effort: S

`MapEditArgs` passes the LLM-emitted `source` string through `ResolveEditSource` (line 338: returns `s` verbatim) into `PendingGenStore.Enqueue` as `initImage`. When the coordinator calls `BuildGenRequest`, this unvalidated path becomes `-InitImage <attacker-path>` in the pwsh argv array. Although there is no shell injection (argv is correctly separated), `doki-gen.ps1` opens the file at that path — meaning an LLM-suggested `../../passwords.txt` causes pwsh to read and upload an arbitrary file to SwarmUI.

**Fix:** In `RunEditImage`, validate `resolved` through `GalleryService.Resolve` before queuing. `GalleryService.Resolve` already performs canonical-path prefix checks and extension allowlisting, giving the same guarantee as `/api/gallery/media/{name}`. Return an error string to the LLM if the path does not resolve.

---

**[P1-3] GenRequest missing `Audio` and `Engine` fields — InfiniteTalk, LatentSync, Speak silently broken from panel**
`control/Services/GenArgs.cs:15` | Effort: S

`doki-gen.ps1` requires `-Audio` for `-InfiniteTalk` (line 527) and `-LatentSync` (line 532), and `-Engine` to select the TtsSuite workflow for `-Speak` (line 223). Neither field exists in `GenRequest` nor is emitted by `GenCli.BuildArgs`. Panel-driven calls for these three kinds throw at the PowerShell gate on a missing mandatory parameter; Speak falls back to IndexTTS2 with no engine override possible.

**Fix:** Add `string? Audio = null` and `string? Engine = null` to the `GenRequest` record. In `BuildArgs`, emit `-Audio` gated on `Kind` being `infinitetalk`/`latentsync`/`speech`, and `-Engine` for speech when non-null. Cover both in `GenCliTests`.

---

**[P1-4] `start-embed.ps1` sets `CUDA_VISIBLE_DEVICES=-1` in the parent process**
`serving/start-embed.ps1:15` | Effort: S

Line 15 mutates `$env:CUDA_VISIBLE_DEVICES` in the calling scope. When `doki.ps1` invokes this script, the assignment persists into the parent session and is inherited by every subsequent `Start-Process` call (TTS, STT, etc.), causing those services to start without GPU access.

**Fix:** Pass the variable as part of `Start-Process`'s `-Environment` parameter hash for the detached path: `$sp.Environment = @{ CUDA_VISIBLE_DEVICES = '-1' }`. For any foreground path, wrap in a `try`/`finally` that restores the previous value.

---

**[P1-5] Model downloads have no SHA-256 verification before promoting `.part` files**
`setup.ps1:582` | Effort: M

`Get-Model` and `Fetch-Gguf` promote on `curl` exit 0 alone. A CDN compromise, BGP hijack, or HuggingFace account takeover yields a silently-accepted malicious weight file. The SAM checkpoint uses bare `Invoke-WebRequest` with no temp-then-move and no hash check. One file (GLM-4.7-Flash) documents a SHA-256 in a comment but never verifies it programmatically.

**Fix:** After each successful download, compute `Get-FileHash -Algorithm SHA256 $tmp` and compare against a `$Hashes` hashtable at the top of `setup.ps1`. The GLM entry already has the correct hash — promote it into a check. For SAM, replace `Invoke-WebRequest` with the `curl`/hash/`Move-Item` pattern used elsewhere. Files explicitly marked out-of-band may carry an explicit `# not auto-verified` note.

---

**[P1-6] SwarmGen.TryHandle has zero tests — drives every generation**
`control/Web/SwarmGen.cs:83` | Effort: S

`TryHandle` parses every WS frame from SwarmUI and decides whether a frame is a progress update, error, single image, or image array. It is static, pure, and total — trivially testable — yet has no coverage. A regression silently loses artifacts or hangs the drain loop.

**Fix:** Add `SwarmGenTests.cs` covering: keepalive returns false; `gen_progress` fires `onProgress` with correct percent/preview; error frame sets error and returns true; image frame and images-array frame each extract the correct path; percent normalization (>1 divided by 100, <=1 kept as-is). All cases are GPU-free.

---

**[P1-7] LocalSecurityMiddleware has no tests**
`control/Web/LocalSecurityMiddleware.cs:19` | Effort: S

The sole auth layer for an unauthenticated localhost server has no test coverage. A refactor of `AllowedHosts` or the Origin-check branch could silently widen the attack surface.

**Fix:** Add `LocalSecurityMiddlewareTests.cs` using `DefaultHttpContext` (no Kestrel needed): allowed Host passes; forbidden Host returns 403; POST with no Origin passes (before [P1-1] fix, then update the test after); POST with allowed Origin passes; POST with foreign Origin returns 403.

### P2 — Fix This Sprint

---

**[P2-1] `SaveDataUrl` temp files (init/mask/control) never cleaned up on the success path**
`control/Web/StudioHost.cs:140` / `control/Web/GenerationJobs.cs:141` | Effort: S

Every `POST /api/generate` with init/mask/control images writes to `%TEMP%\dokidex-{tag}-{guid}.{ext}` and never deletes them. The live-canvas path has explicit `TryDeleteLiveTemp` cleanup; the main generate path does not. Sensitive decoded user images accumulate on disk indefinitely.

**Fix:** Track temp paths in a local list inside the `/api/generate` handler. After `jobs.Submit(req)`, widen the existing `TryDeleteLiveTemp` call in `GenerationJobs.RunAsync` to cover non-ephemeral jobs when the path carries the `dokidex-init-`/`dokidex-mask-`/`dokidex-control-` prefix.

---

**[P2-2] SAM segment-click endpoint also leaves temp PNGs indefinitely**
`control/Web/StudioHost.cs:1053` | Effort: S

`POST /api/segment-click` decodes the input to `%TEMP%\dokidex-sam-{guid}.png` and never deletes it, nor its `r.MaskPath` sidecar. An inpainting session with many SAM clicks leaves growing ~1 MB PNGs per click.

**Fix:** Wrap `Sam.SegmentAsync` in a `try/finally` that deletes `imgPath`. Delete `r.MaskPath` after reading its bytes into `b64`.

**[P2-3] Doc-upload validation block copy-pasted four times**
`control/Web/StudioHost.cs:623, 663, 879, 901` | Effort: M

The Content-Length 413 pre-check, `SafeName` guard, char/byte cap, `IngestAsync`/`IngestBinAsync` call, and `KbId`-persistence are repeated verbatim across four endpoint pairs. The four copies have already silently diverged on minor wording.

**Fix:** Extract `IngestTextDoc(kbId, source, text, req, ct)` and `IngestBinaryDoc(kbId, source, req, ct)` returning a discriminated union. Each of the four endpoints calls the appropriate helper and handles only its own `KbId` side-effect. Eliminates ~120 lines.

**[P2-4] `GalleryService` instantiated with `new` in static helpers, bypassing DI singleton**
`control/Web/Chat.cs:370` / `control/Web/ChatTools.cs:390` | Effort: S

`Chat.ResolveImage` and `ChatTools.Run` each call `new GalleryService()` directly, creating a second non-singleton instance despite DI registering it as a singleton at `StudioHost.cs:50`. Any state added to `GalleryService` (cache, `FileSystemWatcher`, write lock) will silently split across instances. `ChatTools.Run` also re-enumerates the gallery directory on every tool hop (up to 4 per turn).

**Fix:** Thread `GalleryService` as a parameter into both static methods. The endpoint lambdas already have DI access and can pass the singleton down.

**[P2-5] SSE streaming loop duplicated between `/chat/stream` and `/chats/{id}/regenerate`**
`control/Web/StudioHost.cs:365, 483` | Effort: M

The SSE frame-write pattern is implemented twice. The regenerate path adds rollback-on-failure logic. Any change to the SSE framing must be applied to both.

**Fix:** Extract `WriteStreamEvents(HttpContext ctx, IAsyncEnumerable<StreamEvent> events, CancellationToken ct)`. The `/regenerate` endpoint wraps its truncate/restore logic around `Chat.StreamAsync` and passes the result in; the frame format lives in one place.

**[P2-6] Chat.SendAsync / AgentAsync / StreamAsync have a 7-step setup prologue each**
`control/Web/Chat.cs:97, 196, 292` | Effort: M

All three send paths independently execute: validate message, load persona, load-or-create conversation with `ApplyDefaultKb`, `SelectHistory`, `ActivateLore`, `RetrieveDocs`, `MemoryRecall.RetrieveAsync`. Any new context-injection tier must be threaded into three places.

**Fix:** Extract `PrepareAsync(ChatRequest, ...) -> ChatContext` encapsulating steps 1–7. All three paths call `Prepare()` then diverge only on LLM call strategy.

**[P2-7] `setup.ps1` pip installs for gated nodes suppress output and skip `$LASTEXITCODE` check**
`setup.ps1:883` | Effort: S

`WanVideoWrapper`, `LatentSync`, and `InstantID` all use `& $cpy -m pip install ... | Out-Null` with no exit code check. A broken pip install (missing C++ compiler, resolver conflict) passes silently. The sentinel file is not written for these paths, producing a partial install that appears successful.

**Fix:** Replace bare pip calls with the existing `Pip` helper function (defined at line 68), which already checks `$LASTEXITCODE` and throws on failure.

**[P2-8] Chatterbox config.yaml patch silently re-encodes file on round-trip**
`setup.ps1:205` | Effort: S

`Get-Content`/`Set-Content` round-trips the YAML without specifying encoding, silently converting BOM, CRLF, or non-ASCII content. PyYAML is sensitive to encoding marks.

**Fix:** Use `[System.IO.File]::ReadAllText` / `WriteAllText` with explicit `Encoding.UTF8`, matching the pattern already used for `Settings.fds` at line 514 and `Installation.cs` at line 477.

**[P2-9] antelopev2 Expand-Archive may double-nest the .onnx files**
`setup.ps1:826` | Effort: S

If the upstream zip contains a top-level `antelopev2/` folder, files land at `insightface\models\antelopev2\antelopev2\*.onnx`. The sentinel check passes (the file exists somewhere deeper) but the InstantID node fails at runtime when it expects files at the immediate child level.

**Fix:** After `Expand-Archive`, check for `$insDir\glintr100.onnx`. If absent, check one level deeper and `Move-Item` the contents up. Add a comment documenting the expected flat structure.

**[P2-10] `memory_db.py` LIKE fallback does not escape `%` and `_` wildcards**
`serving/memory-mcp/memory_db.py:47` | Effort: S

The FTS5 `MATCH` query is parameterized (SQL injection not possible), but on `sqlite3.OperationalError` the fallback uses `%{query}%` in a LIKE clause without escaping. A stored memory containing `%` matches every row.

**Fix:** Before the LIKE fallback: `escaped = query.replace('%', r'\%').replace('_', r'\_')` with `ESCAPE '\\'`. For FTS5 robustness, phrase-quote the query string as `'"' + query.replace('"','""") + '"'` to prevent malformed expressions.

**[P2-11] `start-tts.ps1` / `start-stt.ps1` foreground path uses `Set-Location`, mutating caller's CWD**
`serving/start-tts.ps1:28` / `serving/start-stt.ps1:25` | Effort: S

The `else` (foreground) branch calls `Set-Location $server` before launching Python, silently changing the CWD for any parent script. `start-kokoro.ps1:50` has the same issue.

**Fix:** Use `-WorkingDirectory` on `Start-Process`, or wrap the `cd` in `Push-Location`/`Pop-Location`.

**[P2-12] `StripAnsi` error-display path for all failed generations is untested**
`control/Web/GenerationJobs.cs:183` | Effort: S

`StripAnsi` is called on every failed generation output. A bug in its multi-byte CSI handling produces garbled error cards for every failure. The function is pure and total with no test coverage.

**Fix:** Extract to `internal`, add 5 cases: plain string passes through; `ESC[31m...ESC[0m` strips; multi-digit CSI (`ESC[1;32m`) strips; all-ANSI string returns empty; null/empty returns empty.

**[P2-13] `SaveDataUrl` base64 decode and temp-file creation have no tests**
`control/Web/StudioHost.cs:141` | Effort: M

`SaveDataUrl` silently returns `null` on any exception (truncated payload, unwritable path), making a legitimate error and a path error indistinguishable to the caller. No test exercises it.

**Fix:** Extract as `internal static (string? v, string tag) -> string?`. Test: null/empty passes through; valid PNG data URL writes `.png`; valid WEBP writes `.webp`; malformed base64 returns null without throwing.

**[P2-14] Eval gate exits 0 regardless of pass rate — model promotions rely on manual inspection**
`evals/run-suite.ps1:1` | Effort: S

`decisions.md` records a 14/15 (93%) threshold as policy but no code enforces it. A 53% run produces a green CI result requiring human intervention to catch.

**Fix:** Add `-MinPassRate` parameter (default 0 for backward compat). At the bottom: `if ($MinPassRate -gt 0 -and $rate -lt $MinPassRate) { Write-Error "GATE FAIL"; exit 1 }`. Call it with `-MinPassRate 87` from the promotion workflow.

**[P2-15] gpt-oss-20b promoted to 'reasoning' tier without clearing >=91% golden-task gate**
`serving/llama-swap.yaml:131` | Effort: M

`decisions.md` explicitly states "The >=91% golden half is UNMEASURED." The model is live and user-selectable on all chat/stream/agent endpoints. Quality failures in multi-step coding look like correct tool-call traces with wrong answers — harder to detect than a flaked tool call.

**Fix:** Run `evals/run-suite.ps1 -Harness crush -Model fast-candidate-gptoss20b` across all 15 tasks. Gate the `reasoning` tier rename on >=91% (14/15), consistent with the coder-fast bar.

**[P2-16] /api/capabilities and /api/home each run a full StatusProbe (nvidia-smi + HTTP) per request**
`control/Web/StudioHost.cs:94, 297` | Effort: M

Every SPA home-page load triggers one or two full probes: nvidia-smi subprocess, two HTTP calls to llama-swap. At 300–800 ms per probe, this is the dominant latency on every home render. The panel's existing 2-second `PollLoop` already does this work.

**Fix:** Cache `StatusDoc` in `DokiService` with a 2-second TTL (matching the poll interval). All three probe sites (`/api/status`, `/api/home`, `/api/capabilities`) read from the cache. Alternatively, expose a lightweight `/api/llama-swap/models` endpoint (one HTTP call, no nvidia-smi) for the capability tier list.

### P3 — Address When Convenient

---

**[P3-1] GPU group mutual-exclusion check has a race window between stop and start**
`doki.ps1:217` | Effort: M

Between the last `StopSvc` and the first `StartSvc` for the new group, a racing SwarmUI auto-restart can respawn a stopped service. `IsRunning` in `StartSvc` would detect it as "already up (untracked)" and skip stopping it, leaving two opposite-group services co-occupying the GPU.

**Fix:** After the stop loop, add a 1–2 pass verification sweep before starting the first wanted-group service and re-issue `StopSvc` for any still-live opposite-group member. Document the residual race.

---

**[P3-2] Always-on polling timers run unconditionally regardless of active view**
`control/Web/wwwroot/index.html:2737` | Effort: S

`pollStatus` (every 2s) and `pollJobs` (every 1.5s) run from page load to tab close even on views that use neither (Memory, Models, Director). The chat-gen poll already uses a start/stop pattern.

**Fix:** Apply the same `startChatGenPoll`/`stopChatGenPoll` pattern to both timers. Gate `pollJobs` on views with a Results grid. Add a `document.visibilitychange` guard to suspend all polls when the tab is hidden.

---

**[P3-3] `llama-swap.yaml` global `healthCheckTimeout: 900` blocks fast swaps**
`serving/llama-swap.yaml:10` | Effort: M

A 15-minute health timeout was set for `coder-big`'s ~60 GB page-in. It applies equally to `coder-fast` (loads in seconds). A stalled `coder-big` health check can delay `coder-fast` swaps.

**Fix:** If llama-swap v224 supports per-model `healthCheckTimeout`, add a 60s override for `coder-fast` and `coder-fast-lite`. Otherwise document the mismatch and consider a separate llama-swap instance for `coder-big`.

---

**[P3-4] `TailLogs` position tracker races a log rotation**
`doki.ps1:329` | Effort: S

`(Get-Item $f).Length` is read before opening the file. A `RotateLog` between the two operations causes the next iteration to seek past the beginning of a new file, silently skipping new content. The `catch {}` at line 347 swallows the `IOException`.

**Fix:** Read `FileStream.Length` after opening, not before. If `stream.Length < $pos[$f]`, reset `$pos[$f] = 0` (file was replaced). Remove the swallowing `catch`.

---

**[P3-5] `setup.ps1` `$PSNativeCommandUseErrorActionPreference = $false` masks exit codes globally**
`setup.ps1:35` | Effort: M

Necessary for winget's non-standard success codes, but means `dotnet build`, `git checkout`, and several `pip install` calls can fail silently. The global setting cannot be changed; individual calls need guards.

**Fix:** Add explicit `if ($LASTEXITCODE -ne 0) { throw '...' }` after `dotnet build` (line 417), `git -C $swarm checkout` (line 447), and any unguarded native call where failure matters.

---

**[P3-6] Chat tool executors block ASP.NET threads via `.GetAwaiter().GetResult()`**
`control/Web/ChatTools.cs:414` | Effort: M

`RunWebSearch` and `RunCodeSearch` block thread-pool threads for up to 20–30 seconds during sidecar timeouts. Under concurrent agent-loop load this can exhaust the ASP.NET thread pool.

**Fix:** Make `ChatTools.Run` async (`RunAsync` returning `Task<string>`) and await sidecar calls. `AgentAsync` already drives an async loop; threading async through is the only change required.

**[P3-7] Vision tiers missing `--cache-reuse` and quantised KV**
`serving/llama-swap.yaml:72, 87` | Effort: S

Both `vision` (Qwen3-VL-32B) and `vision-8b` omit `--cache-reuse`, paying full prefill cost on every Describe/Verify call even when the system prompt is identical. `vision-8b` also defaults to F16 KV, using ~3.2 GB vs ~1.6 GB for q8_0. The existing `coder-fast` (same Qwen family) uses q8_0 KV without issue.

**Fix:** Add `--cache-reuse 256` to both vision entries. Add `--cache-type-k q8_0 --cache-type-v q8_0` to `vision-8b`. For the 32B tier, verify VRAM budget before enabling KV quantisation (mmproj F16 + q8_0 KV at 32k is tight on 32 GB).

**[P3-8] gpt-oss-20b F16 KV at 131k context costs ~20 GB KV alone**
`serving/llama-swap.yaml:137` | Effort: S

F16 KV at 131072 context for a 20B model costs roughly 20 GB, pushing total VRAM past 32 GB when combined with MXFP4 weights (~18 GB). The reasoning use-case is single-turn chain-of-thought, not retrieval-augmented multi-turn.

**Fix:** Reduce `-c` to 32768 or 65536. At 32k, F16 KV costs ~5 GB, keeping total VRAM comfortably under 32 GB. If 131k context is genuinely needed, benchmark q4_0 KV before falling back to F16.

**[P3-9] `coder-big` `--n-cpu-moe 22` has never been benchmarked**
`serving/llama-swap.yaml:55` | Effort: M

The comment says "tune DOWN for more GPU = faster" but there is no record of this tuning. At 22 offloaded layers, a significant fraction of each forward pass crosses the PCIe bus.

**Fix:** Run `llama-bench` with `--n-cpu-moe` at {22, 16, 12, 8}. Stop when VRAM exceeds ~30 GB. Expected: 30–60% generation throughput improvement per step moved onto GPU.

**[P3-10] `CapabilityCatalog.LoadFromShellAsync` spawns pwsh on first `/api/capabilities` request**
`control/Web/CapabilityCatalog.cs:68` | Effort: S

The shell path adds 30s of potential cold-start latency. `KindSyncTests` already enforces that the static fallback always matches `doki-gen.ps1`'s output, making the shell path redundant.

**Fix:** Remove the shell load path entirely and always return `StaticFallback`. If dynamic kinds are ever genuinely needed, warm them up in a background `Task` at server startup, never on the first request.

**[P3-11] Static store classes have no injectable seam for test isolation**
`control/Web/ChatStore.cs:38` | Effort: M

All store classes resolve disk paths via `RepoPaths.Root` at call time. Tests set this to a temp directory, which is a fragile cross-process global — parallel test classes can collide on the same paths.

**Fix:** Introduce `IStoreRoot` with a single base-directory property. Production DI provides the real root; each test class instantiates its own isolated temp directory.

**[P3-12] AgentAsync 4-minute wall-clock budget timeout path is untested**

The `turnCts` linked token source is the safety valve against stacked LLM timeouts, but the timeout-fires-mid-hop branch (returns graceful fallback, persists both turns) has no test.

**Fix:** Add a test that stubs `LocalLlm.ChatToolsAsync` to delay past a very short `AgentTurnTimeout` override, asserting: method returns `Ok=true` with non-empty graceful message; turn persists to `ChatStore`.

## Strategic Work

These require sustained effort but have the highest architectural payoff:

**Split `StudioHost.MapApi` into per-domain extension methods (Effort: L)**
`control/Web/StudioHost.cs:75`

The 1,170-line `MapApi` method is the most leveraged refactor in the codebase. Extract per-domain files under `Web/Endpoints/`: `GenerationEndpoints.cs`, `ChatEndpoints.cs`, `GalleryEndpoints.cs`, `KbEndpoints.cs`, `ToolsEndpoints.cs`, `MediaEndpoints.cs`. Extract `DataUrlDecoder` as a static utility. Extract `SseWriter` shared by `/chat/stream` and `/regenerate`. This is a pure structural refactor with no behavior change — the existing test suite stays green throughout. Doing this before adding the chat-media round-trip UI will save weeks of navigation cost on every subsequent endpoint.

**Migrate `index.html` to a modular build (Effort: L)**
`control/Web/wwwroot/index.html:1`

A 2,741-line monolithic SPA with 12 co-resident views, all global state, and two always-on polling loops will become load-bearing technical debt the moment the next major view (chat-media round-trip) is added. Options in increasing scope: (a) short-term: extract the 155-line CSS to `index.css`, reorganize JS into per-view IIFE sections with clear demarcation — minimal, reversible; (b) full: Vite + vanilla TypeScript, one module per view, per-view state, StudioHost's `LoadIndex` unchanged (build output is still a single embedded file). Given the chat-media round-trip work described in memory, this is worth prioritizing before that feature begins.

**SHA-256 verification for all auto-downloaded model weights (Effort: M)**
`setup.ps1:582`

**Harden the ASP.NET thread model by making `ChatTools.Run` async (Effort: M)**
`control/Web/ChatTools.cs:414`

The sync-over-async `GetAwaiter().GetResult()` in tool dispatch is the one architectural decision that can cause thread-pool exhaustion under concurrent agent-loop load. Making `Run` async is not a large change — `AgentAsync` already drives an async loop — but it requires threading `async` through a few layers. This becomes more urgent as the agent-loop hop limit increases or as more sidecar-backed tools are added.

**Introduce `IStoreRoot` for test isolation (Effort: M)**
`control/Web/ChatStore.cs`

The cross-process global `RepoPaths.Root` in every store class is a low-grade test reliability issue today but will become a concrete problem when the test suite grows to cover coordinator-level flows that exercise multiple stores. Introducing `IStoreRoot` is a pure extraction — no on-disk format changes, no endpoint behavior changes — and enables parallel test execution without collision.

## What's Genuinely Well-Built

**No shell injection anywhere.** The entire pwsh shelling path uses `ProcessStartInfo.ArgumentList` (proper argv array) throughout `GenCli`, `Sam`, `DokiService`, and all start scripts. LLM-generated values that reach the process launcher are never string-concatenated into command lines. This is the right discipline and the audit found no exceptions.

**DNS-rebinding defense is correct.** `LocalSecurityMiddleware` correctly validates the `Host` header against an allowlist before the Origin check. The CSRF finding (P1-1) is an omission on the no-Origin branch, not a flaw in the Host-header logic itself.

**Pure function extraction is consistent.** `ShouldContinue`, `FilterRecent`, `BuildGenRequest`, `NeedsFlip`, `AppendToolRound`, `SelectHistory`, `ActivateLore` — the team consistently pulls decision logic out of I/O-bound callers and into testable pure functions. The 563+ Fact/Theory suite reflects this discipline.

**Gallery path traversal is properly scoped.** `GalleryService.Resolve` performs a canonical-path prefix check and extension allowlist that is correctly wired into all gallery media endpoints. The `edit_image` gap (P1-2) is the one place this protection is not applied, not a general pattern failure.

**Graceful-degradation contracts are real.** Every service probe path falls back gracefully: `StatusProbe` returns a degraded `StatusDoc`, `CapabilityCatalog` has a `StaticFallback`, the eval harness degrades to pass rather than crash on a missing model. These contracts are tested and held.

**The eval harness is a genuine asset.** A 15-task golden suite with injection tests, hidden test layers, and a results ledger is significantly more rigorous than most single-developer projects. The process around it (recording decisions, noting gaps explicitly) shows the right instincts — the gaps (P2-14, P2-15) are about enforcement, not about the harness design.

**Multipart form size cap is correctly configured.** The `RequestSizeLimitBytes` configuration in the doc-upload endpoints prevents unbounded memory allocation from adversarial uploads — correctly placed at the endpoint level, not globally.

**The WPF layer is well-structured.** `MainViewModel`, `StudioViewModel`, and the relay-command pattern are clean and do not exhibit any of the god-class tendencies of the web layer. No urgent attention needed there.
